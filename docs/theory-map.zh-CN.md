# 理论地图 · Product Route 的思考框架

## 为什么需要 Product Route？

大多数产品想法评估的问题在于：它们问的是「这个想法好不好」，而不是「接下来该做什么」。

「好不好」是一个无法回答的问题 — 它依赖太多外部因素：时机、执行、市场变化、运气。但「接下来该做什么」是一个可以回答的问题 — 它依赖的是**证据**和**模式识别**。

Product Route 的核心洞察是：**不同的产品想法需要完全不同的推进模式**。一个二维码生成器和一个视频托管平台不应该被放在同一个评价框架里比较。

---

## 三条轴线

Product Route 的路由体系基于三条隐含轴线：

### 轴线 1：交付复杂度 × 责任风险

这决定了产品的「安全区」：

| | 低责任风险 (S09 <= 3) | 高责任风险 (S09 >= 4) |
|---|---|---|
| **低交付复杂度 (S08 <= 2)** | R01 入口工具 | R04 第三方封装 |
| **中交付复杂度 (S08 = 3)** | R02 货架服务 | R04 / R05 销售型 |
| **高交付复杂度 (S08 >= 4)** | R03 内部基础设施 | R07 高风险暂缓 |

### 轴线 2：付费意愿 × 获客路径

这决定了产品的「商业模式」：

| | 弱获客 (S07 <= 2) | 强获客 (S07 >= 3) |
|---|---|---|
| **弱付费 (S05 <= 2)** | R03 内部基础设施 / R09 先调研 | R01 入口工具 |
| **强付费 (S05 >= 3)** | R05 销售型服务 | R02 货架服务 |

### 轴线 3：证据强度 × 风险等级

这决定了评估的「置信度」：

| | 低风险 | 高风险 |
|---|---|---|
| **强证据 (A/B)** | 可以直接推进 | 窄化后推进（R04） |
| **弱证据 (C/D)** | R09 先调研 | R07 暂缓 + R09 调研 |

---

## 路由之间的关系

路由不是互斥的分类，它们之间存在演化路径：

```
R09 (调研) ──→ R01 (入口工具) ──→ R02 (货架服务) ──→ R05 (销售型服务)
                  │                      │
                  └──→ R03 (内部)        └──→ R04 (封装)
                                          │
R07 (高风险暂缓) ──→ R04 (第三方封装) ──→ R02 (货架服务)
```

一个产品想法可以在不同阶段走不同的路由。但**一次评估只能给出当前阶段的推荐路由**。

---

## 什么不是 Product Route 的关注点

Product Route 刻意不关注以下维度：

1. **市场规模（TAM）** — 大多数早期产品不需要知道精确市场规模，需要知道的是有没有人愿意用、有没有人愿意付费
2. **团队能力评估** — Product Route 评估的是产品想法本身，不是团队。S12（阶段适配度）留给用户自己判断
3. **投资回报率（ROI）** — Product Route 不是投资工具
4. **竞品功能对比矩阵** — 太细的功能对比会让人误以为在做产品，而不是在验证方向

## 参考阅读

- **Saras Sarasvathy, "Effectuation"** — 强调从现有手段出发，而不是从目标出发
- **Steve Blank, "The Four Steps to the Epiphany"** — 客户开发方法论
- **Basecamp, "Shape Up"** — 限时、限范围的开发理念
- **Taleb, "Skin in the Game"** — 风险和责任不对称
