# 反模式 · Anti-Patterns

> 常见的执行模式错误。这些错误杀死的产品比"坏点子"多得多。

---

## AP01 · 把内容产品做成 SaaS

**错误：** 核心价值是内容/知识/社群，却按 SaaS 模式建付费墙和功能矩阵。

**为什么失败：**
- 用户为内容而来，不为工具而来
- 没有内容分发循环 = 没有留存
- SaaS 的用户获取模型（SEO/广告）不适用于内容驱动产品

**例子：**
- 一个 AI Twitter 摘要工具 → 不应该先做工具，应该先做内容输出
- 一个行业分析平台 → 不应该先做数据面板，应该先做报告

**正确做法：** R06 内容/社区产品。先做内容，验证受众，再叠加工具。

---

## AP02 · 把基础设施当独立产品

**错误：** 把需要专业团队维护的基础设施当成个人/小团队可以做的独立 SaaS。

**为什么失败：**
- 存储、带宽、安全、合规成本线性增长
- 内容审核、版权、滥用是运营黑洞
- 竞品有数百人和数亿美元融资

**例子：**
- 完整视频托管平台 → R07 高风险基础设施
- 完整邮件发送平台 → R07
- 完整支付处理系统 → R07

**正确做法：** R04 第三方封装。选用成熟底层服务，只做管理/配置层。

---

## AP03 · 把封装工具建成平台

**错误：** 用一个第三方 API 的封装起步，然后不断加功能，直到变成复杂系统。

**为什么失败：**
- 封装的价值是简化，不是扩展
- 封装层本身变成复杂系统 = 失去了"简化"的定位
- 底层 API 变更/涨价 = 业务崩溃

**例子：**
- Stripe 支付管理面板 → 不要再自己加支付处理
- Vimeo 嵌入管理 → 不要再做视频转码
- SendGrid 邮件管理 → 不要再做 SMTP 服务器

**正确做法：** R04 保持在封装层。如果确实需要扩展，升级到 R02 货架服务并明确依赖边界。

---

## AP04 · 把免费工具货币化

**错误：** 把用户认为"应该免费"的工具包装成付费 SaaS。

**为什么失败：**
- 用户付费意愿为零不是因为缺少功能，而是因为这是基础设施
- 竞品免费或有操作系统内置
- 强行收费 = 用户流失到免费替代

**例子：**
- 二维码生成器 → R01 入口工具，不是 R02 货架服务
- 图片压缩工具 → R01
- 格式转换工具 → R01

**正确做法：** R01 入口工具。免费，靠分发获客，作为更大产品矩阵的入口。

---

## AP05 · 在零证据下构建

**错误：** 在没有任何外部证据的情况下开始构建完整产品。

**为什么失败：**
- "我觉得有人需要"不是证据
- 没有用户验证的产品几乎总是错的
- 构建成本是沉没成本，但维护成本和机会成本不是

**信号：**
- 说不清具体触发场景（S01 <= 2）
- 说不清目标用户是谁（S06 <= 2）
- 没有竞品或付费信号（S05 = D 级）

**正确做法：** R09 信息不足，先调研。做 3 个 1-3 天的低成本验证动作，收集证据后再评估。

---

## AP06 · 把低频高价值产品判死刑

**错误：** 因为使用频率低（S03 <= 2）就自动判定产品没价值。

**为什么失败：**
- 低频不等于不重要
- 某些产品的价值恰恰在于"平时不需要，需要时没有就完了"
- 这类产品的付费意愿通常不低

**例子：**
- 隐私政策托管 → 审核时才需要，但审核被拒 = 上线失败
- 状态页 → 故障时才需要，但没有状态页 = 客户流失
- 数据删除页 → 合规检查时才需要，但没有 = 合规风险

**正确做法：** R02 货架服务。低频强痛点是最适合货架服务的组合。

---

## AP07 · 把销售驱动产品当自助 SaaS

**错误：** 把需要销售介入、定制接入、合同谈判的产品做成自助购买。

**为什么失败：**
- 客户不会在网站上点"购买"就签几十万的合同
- 自助 SaaS 的定价模型无法支撑销售成本
- 忽视实施、培训、集成的实际成本

**正确做法：** R05 销售型服务。先找 1-2 个付费客户验证客单价和交付成本。

---

## AP08 · 把所有功能当独立产品

**错误：** 把"作为功能很有价值"的东西做成独立产品。

**为什么失败：**
- 用户不会单独为此付费
- 独立产品需要获客、留存、支持 — 成本远超作为功能
- 作为现有产品的功能反而更容易被发现和使用

**正确做法：** R08 只适合做功能。作为开源模块贡献，或集成到现有产品中。
