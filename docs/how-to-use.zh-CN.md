# 使用指南 · Product Route

## 适用场景

Product Route 适用于以下场景：

1. **你有一个产品想法，不确定该怎么做** → Product Route 帮你判断推进模式
2. **你有多个产品想法，需要决定先做哪个** → 但 Product Route 不做横向比较，请对每个想法单独评估
3. **你已经在做产品，想确认方向是否正确** → Product Route 帮你检查是否忽略了风险
4. **你想让 AI Agent 帮你结构化评估产品想法** → 安装本 Skill 后，Agent 会按规则执行

---

## 安装方式

### 方式 A：作为 Skill 包（推荐）

将 `skills/product-route.zh-CN/` 安装到你的 Agent Skill 目录。

安装后，Agent 会在检测到产品想法评估相关意图时自动触发。

### 方式 B：单文件 Bundle

将 `dist/product-route.zh-CN.bundle.md` 作为 System Prompt 注入。

适合不支持 Skill 目录结构的 Agent 环境。

### 方式 C：程序化调用

将 `schemas/result.schema.json` 集成到你的产品决策 pipeline 中。

## 使用示例

### 示例 1：评估一个新想法

```
用户：帮我看看这个想法 — 做一个给独立开发者用的 CI/CD 面板，自动部署到 VPS

Agent（使用 Product Route）：
→ 复述想法 → 建立证据表 → 评分 → 路由判断
→ 主路由：R03 内部基础设施（高复用价值，但获客不清晰）
→ 建议先满足自己需求，验证后再考虑对外
```

### 示例 2：检查现有产品的风险

```
用户：我做了 3 个月的支付管理面板，可以自动处理退款和账单，你觉得应该继续吗？

Agent（使用 Product Route）：
→ 识别 S09 责任风险 >= 4 + S08 交付复杂度 >= 4
→ 触发 G01 支付与资金风险门槛
→ 主路由：R07 高风险基础设施，暂缓
→ 副路由：R04 第三方封装（建议使用 Stripe 作为底层）
```

## 理解输出

### 主路由和副路由

- **主路由**是当前阶段的核心推荐，决定了推进模式和 MVP 范围
- **副路由**是备选方案或可演化的方向

### 置信度

- **A**：证据充足，判断可信度高
- **B**：关键证据基本充足，判断可信
- **C**：部分证据缺失，判断有一定不确定性
- **D**：关键证据大量缺失，建议先调研

### 不做清单

不做清单和做清单一样重要。Product Route 的一个核心原则是：**低成本的窄化方案优于高风险的完整产品**。

## 常见问题

### Q：Product Route 能告诉我该不该做这个产品吗？

不能。Product Route 不判断「该不该做」，它判断「如果要做，应该用什么模式推进」。

### Q：评分高就一定好吗？

不。有些维度（S08、S09、S10）分数越高代表风险越大。路由判断基于维度组合，不是总分。

### Q：为什么需要证据？我觉得自己判断就可以了。

没有证据的评估几乎总是高估痛点的强度和付费意愿。Product Route 强调「先证据，后评分」，证据不足时降置信度，防止在信息不足时做出代价高昂的决策。

### Q：外部案例库有什么用？

外部案例库（Case Library / Competitor Provider / Keyword Provider）可以提供更高质量的证据，提高置信度。但它们是可选的 — 核心 Skill 在没有外部服务的情况下也能独立运行。
