# 示例评估：在线思维导图工具

## 产品想法

做一个在线思维导图工具，支持实时协作、Markdown 导入导出、AI 辅助生成节点。用户可以在浏览器中创建、编辑、分享思维导图，多人同时编辑，从 Markdown 文本自动生成导图结构，并通过 AI 根据主题自动扩展子节点。

---

## 1. 产品想法复述

> 用一句话复述产品想法，确认理解一致。

**想法：** 做一个浏览器端的思维导图工具，核心卖点是多人实时协作编辑、Markdown 双向转换、以及 AI 根据用户输入的主题自动生成和扩展思维导图节点。

---

## 2. 目标用户

- 知识工作者：产品经理做需求拆解、技术方案设计
- 学生：课程笔记整理、考试复习提纲
- 项目经理：项目规划、WBS 分解
- 内容创作者：文章大纲、课程结构设计

但注意：以上群体均已高度分散在 Miro、Notion、Obsidian、Xmind 等现有工具中，难以精准定向触达。

---

## 3. 触发场景

- 需要快速把零散想法整理成结构化思维导图（如会议后的头脑风暴整理）
- 团队远程协作进行项目规划或方案讨论，需要多人同时编辑同一张导图
- 已有 Markdown 格式的笔记或大纲，希望一键转换成可视化思维导图
- 拿到一个主题（如"新产品功能规划"），希望 AI 帮助拆解成完整的思维导图框架

触发事件共性：知识结构化需求。但触发事件缺乏外部驱动力（不是审核、合规、故障等强触发），属于用户"有时间就可以做"而非"必须马上做"的场景。

## 9. 路由判断

**主路由：** R08 — 只适合做功能，不适合独立产品（Feature Only, Not Standalone）

**副路由：** 无

**命中的路由规则：**

- **手动判断（核心逻辑）：** S04 = 5（成熟竞品密布：Miro、FigJam、Excalidraw、Xmind、Lucidchart、MindMeister）+ S05 ≤ 2（用户不会为独立的思维导图工具付费）→ 作为独立产品用户感知价值不足，但作为笔记应用/知识库/白板工具的内置功能非常有价值 → R08
- 硬性规则中，无规则直接命中 R08（R08 当前无硬性触发条件，依赖综合判断）
- 硬性规则排除：S09 < 4 且 S08 限制后 = 3 → 不触发 R07；S05 ≤ 2 但 S07 ≤ 2 → 不触发 R01 的"弱付费 + 强获客"规则；S06 = 3 > 2 → 不触发 R09；S01 = 3 > 2 → 不触发 R09

**为什么不是其它路线：**

- **R01 入口工具：** 核心差异在于"分布价值"。QR 码生成器之所以是 R01，是因为大量用户主动搜索"QR code generator"进入你的产品，你有机会向他们展示其它工具，形成流量入口效应。思维导图工具不具备这一特性 — 用户不会通过搜索"思维导图"发现你的产品生态，且使用完思维导图后没有自然承接的下一步产品。S07 = 2（获客路径不清晰），硬性规则"S05 ≤ 2 且 S07 ≥ 4 → R01"不满足（S07 = 2，远低于 4）。思维导图是"目的地产品"而非"入口产品"。

- **R02 货架服务：** S05 = 2（付费意愿弱），硬性规则"S04 ≥ 4 且 S05 ≥ 3 且 S08 ≤ 3 → 可考虑 R02"不满足（S05 不达标）。用户不会为独立的思维导图工具在货架上自主购买 — 他们已经在 Miro 或 Notion 的付费订阅中免费获得了这个功能。低频（S03 = 2）且付费意愿弱 → 不是货架服务。

- **R03 内部基础设施：** S11 = 3（复用价值中等），硬性规则"S11 ≥ 4 且 S07 ≤ 2 → R03"不满足。虽然 S07 低，但复用价值不足以支撑作为内部基础设施。且S06 = 3（用户明确），有明确的潜在外部用户，不是纯内部工具。

- **R04 第三方封装：** 底层没有需要封装的高风险第三方服务（S09 = 2）。这个产品的核心价值（实时协作、AI 节点生成）恰恰是需要从零构建的部分，没有成熟的第三方可封装。不是"封装 Miro API"或"封装 Xmind API"的场景。

- **R05 销售型服务：** 思维导图工具是典型的自助使用产品，不需要销售介入、定制接入或合同谈判。客单价低，不适合销售驱动模式。

- **R06 内容或社区产品：** 核心价值是工具功能（思维导图编辑、协作、AI 生成），不是内容或社群关系。如果围绕"思维导图模板社区"或"学习方法论内容"来建，那才是 R06，但当前想法是以工具为核心。

- **R07 高风险基础设施，暂缓：** S09 = 2（责任风险低），硬性规则"S09 ≥ 4 且 S08 ≥ 4 → R07"不满足。不涉及支付、认证、邮件、视频等高基础设施风险。

- **R09 信息不足，先调研：** S01 = 3 > 2，S06 = 3 > 2，关键维度中无 D 级证据。信息足够做出判断（R08），不需要先调研。

## 10. MVP 边界

> 注意：按 R08 路由，不建议独立建 MVP。以下边界描述的是"如果作为某个现有产品的功能模块来开发"时的范围。

**第一版只做什么：**

1. 基础的思维导图节点创建、编辑、拖拽排列
2. Markdown 文本粘贴后自动解析为思维导图结构
3. 导出为 PNG/SVG 图片格式
4. 作为现有产品（如笔记应用）的一个视图模式嵌入

**明确不做什么：**

1. 不做独立的用户注册和登录系统
2. 不做独立的网站、独立域名、独立品牌
3. 不做实时多人协作（作为功能模块时，复用宿主产品的协作基础设施）
4. 不做 AI 自动生成节点（V2 再考虑，作为宿主产品 AI 能力的一部分）
5. 不做付费订阅体系

**依赖哪些第三方服务：**

| 服务 | 用途 | 替代方案 |
|---|---|---|
| 宿主产品的认证系统 | 用户身份管理 | — |
| 宿主产品的存储服务 | 导图数据持久化 | — |
| 开源思维导图渲染库（如 markmap、mind-elixir） | 导图渲染引擎 | D3.js 自绘 |

**不承诺哪些责任：**

1. 不承诺作为独立产品提供服务
2. 不承诺在宿主产品之外的兼容性

## 11. 不做清单

> 基于 R08 路由：这些是在当前阶段明确不应该做的事情。

1. ❌ 不做独立的商业网站和独立品牌推广 — 作为独立产品没有市场竞争力，应该作为现有产品的功能模块
2. ❌ 不做独立的用户体系（注册、登录、付费、团队管理）— 复用宿主产品的用户系统
3. ❌ 不做从零开发的实时协作引擎（CRDT/OT）— 如确需协作，复用宿主产品的协作基础设施或使用 Liveblocks/PartyKit 等成熟服务
4. ❌ 不做独立的 AI 节点生成服务 — AI 差异化窗口极短，不值得投入独立开发，应复用宿主产品已有的 AI 能力
5. ❌ 不做移动端原生 App — 独立思维导图 App 与 Xmind 移动版正面竞争，毫无胜算
6. ❌ 不做思维导图模板市场 — 属于 R06 内容社区路线，偏离核心判断
7. ❌ 不做 API 对外开放 — 独立思维导图 API 没有付费用户基础

## 12. 最低成本验证动作

> 3 个具体动作，每个能在 1-3 天内完成。

### 动作 1：竞品功能矩阵对比

**目标：** 用数据确认替代方案成熟度（S04 = 5），并找到"作为功能"的切入点

**具体步骤：**
1. 列出 Miro、FigJam、Excalidraw、Xmind、Notion、Obsidian 的思维导图相关功能矩阵（创建/编辑/协作/导出/AI/模板）
2. 标注每个竞品的免费版限制和付费墙位置
3. 在至少 2 个竞品的用户社区（Reddit、Discord、Product Hunt 评论区）搜索"mindmap"关键词，收集用户抱怨和功能请求
4. 输出：哪些竞品缺少什么？哪个缺口可以作为"功能模块"的切入点？

**预计时间：** 2 天

---

### 动作 2：确认"应该内嵌到哪个产品"的假设

**目标：** 验证"思维导图更适合作为笔记/知识库功能"这一判断，找到最合适的宿主产品类型

**具体步骤：**
1. 在 Product Hunt、Hacker News、Reddit r/Notion、r/ObsidianMD 搜索"mindmap feature request"相关帖子
2. 统计哪类产品的用户最常请求内置思维导图功能（笔记工具？项目管理工具？白板工具？）
3. 记录至少 5 条真实用户的原话，标注来源（帖子链接）
4. 输出：用户最希望在哪个类型的现有产品中看到思维导图功能？

**预计时间：** 1.5 天

---

### 动作 3：用开源库快速验证渲染可行性

**目标：** 验证技术可行性，确认交付复杂度（S08）的真实水平，同时产出一个可嵌入的 Demo

**具体步骤：**
