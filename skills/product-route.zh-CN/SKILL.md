# Product Route Skill · 中文运行包

> runtime_language: zh-CN
>
> **语言隔离：** 本运行包只加载中文规则。不得在同一 Agent 上下文中同时安装 `product-route.en`。仓库可以双语，运行必须单语。

## When to Use

当用户讨论产品想法时使用本 Skill。关键信号 — 用户在问"怎么做"而不是"好不好"：

**明确触发：**
- 评估产品想法、产品方向、产品定位
- 判断推进模式（工具 / SaaS / 内部设施 / 内容 / 销售驱动 等）
- 定义 MVP 边界
- 识别高风险基础设施
- 判断独立产品 vs 功能模块

**口语化触发（容易漏判，特别注意）：**
- 「这个想法靠谱吗」「能做吗这个」「该不该做」
- 「帮我看看这个方向」「帮我分析下这个产品」
- 「我是个人开发者，这个能做吗」
- 「这玩意儿有市场吗」「有人会付费吗」
- 「和 xxx 比有什么优势」「为什么没人做这个」
- 「我想做一个 xxx，你觉得怎么样」

## When NOT to Use

- 法律意见、合规审查
- 财务建议、投资决策
- 精确市场规模预测
- 商业计划书撰写
- 在没有任何证据的情况下鼓励用户立即开发
- 替代用户做最终决策

## Required Process

每次评估必须严格按以下步骤执行，不可跳过：

1. **复述产品想法** — 用一句话确认你理解的产品想法
2. **建立证据表** — 列出当前已知的所有证据，标注类型和来源
3. **建立反证表** — 主动寻找反面证据和风险信号
4. **12 维评分** — 读取 `references/scoring-rules.md`，逐项评分
5. **置信度惩罚** — 读取 `references/evidence-rules.md`，对证据不足的维度降分
6. **风险门槛检查** — 读取 `references/risk-gates.md`，检查是否触发 G01-G08
7. **路由判断** — 读取 `references/routing-rules.md`，决定主路由和副路由
8. **输出报告** — 按 `references/report-template.md` 格式输出完整报告

## Required References

评估过程中必须读取以下文件：

- `references/scoring-rules.md` — 12 维评分体系（S01-S12）
- `references/routing-rules.md` — 9 种路由规则（R01-R09）含硬性路由规则
- `references/evidence-rules.md` — 证据等级（A-D）+ 惩罚机制 + MCP/Provider 降级
- `references/risk-gates.md` — 8 个风险门槛（G01-G08）
- `references/report-template.md` — 14 章节固定输出模板

## Hard Rules（硬性规则）

1. **先证据，后评分。** 人在评估自己想法时几乎总是高估痛点强度和付费意愿。没有外部验证的"我觉得"会把一个应该走 R09 调研的想法误判为 R02 货架服务。证据先行是为了对抗确认偏误。

2. **没有证据不得给高分。** 任何维度若无 A/B 级证据，最高只能打 3 分；若只有 D 级证据（纯灵感），最高只能打 2 分。这是为了防止在信息真空里做决策。

3. **不得只用总分决策。** 一个二维码生成器和视频托管平台可能在某个维度上得分相同，但它们的执行模式完全不同。路由判断必须基于维度组合和硬性规则匹配，不是加总。

4. **高风险基础设施必须先过 risk gates。** S09 >= 4 且 S08 >= 4 时，必须路由到 R07 或 R04。因为个人开发者/小团队从零造支付、视频、邮件系统不是"难"，是"危险"。

5. **一次评估只允许一个主路由，最多一个副路由。** 给 3 个以上的路由等于没给建议。

6. **自然语言输出中文。** 报告正文使用中文。

7. **JSON key 使用英文。** 结构化输出中的字段名必须是英文。

8. **不得混用英文规则文件。** 本次运行只加载中文规则。仓库双语，运行单语。

## Output Must Include

每份评估报告必须包含 14 个章节：

1. 产品想法复述
2. 目标用户
3. 触发场景
4. 证据表
5. 反证表
6. 12 维评分表
7. 置信度惩罚说明
8. 风险门槛检查结果
9. 路由判断（主路由 + 副路由 + 命中规则 + 排除理由）
10. MVP 边界（做什么 + 不做什么 + 第三方依赖 + 不承诺）
11. 不做清单（至少 5 条）
12. 最低成本验证动作（3 个具体动作，每个 1-3 天）
13. 外部证据缺口
14. JSON Result（key 使用英文）
