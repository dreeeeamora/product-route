## 路由总览

| ID | 名称 | 英文名 | 一句话描述 |
|---|---|---|---|
| R01 | 入口工具 | Entry Tool | 低门槛工具，靠 SEO/分发获客 |
| R02 | 货架服务 | Shelf Service | 标准化 SaaS，用户自助购买，长期挂在货架上 |
| R03 | 内部基础设施 | Internal Infrastructure | 复用型内部工具，先满足自己再考虑对外 |
| R04 | 第三方封装 | Third-party Wrapper | 封装底层复杂服务，提供简化界面 |
| R05 | 销售型服务 | Sales-led Service | 需要销售介入、定制接入、合同谈判 |
| R06 | 内容或社区产品 | Content / Community | 以内容或社群为核心价值，工具是辅助 |
| R07 | 高风险基础设施，暂缓 | High-risk Infrastructure | 涉及高风险，建议窄化或暂缓，不要从零造 |
| R08 | 只适合做功能 | Feature Only | 不适合独立产品，应作为现有产品的功能模块 |
| R09 | 信息不足，先调研 | Research First | 关键证据缺失，先收集信息再做判断 |

## R01 · 入口工具 · Entry Tool

**适合条件：**
- 用户明确，低门槛
- 使用频率高（S03 >= 3）
- 付费意愿弱（S05 <= 2）
- 获客路径清晰（S07 >= 3）
- 交付复杂度低（S08 <= 2）
- 可以作为更大产品矩阵的流量入口

**不适合条件：**
- 付费意愿强（更适合 R02 货架服务）
- 交付复杂（S08 >= 4）
- 责任风险高（S09 >= 4）

**常见例子：**
- 二维码生成器
- 图片压缩工具
- 格式转换工具
- 小计算器 / 检查器
- SEO 关键词工具
- 免费模板

**推荐推进方式：**
- 1-2 周出 MVP
- 靠 SEO / 内容 / 免费分发获客
- 不必做复杂付费体系
- 可作为产品矩阵的入口节点
- 如果后续发现付费信号，可升级为 R02 货架服务

## R02 · 货架服务 · Shelf Service

**适合条件：**
- 痛点强度较高（S02 >= 3）
- 使用频率可以低，但触发场景必须明确
- 如果 S03 <= 2，则 S01 或 S02 至少应 >= 4（低频但强触发或强痛点）
- 用户有一定付费意愿（S05 >= 3）
- 交付复杂度可控（S08 <= 3）
- 责任风险可控（S09 <= 3）
- 可以标准化、自助购买、长期挂在货架上

**补充说明：**
- 货架服务不一定高频。低频强痛点非常适合货架服务。
- 典型低频货架服务：政策页托管（审核时触发）、状态页（故障时触发）、产品更新日志（发布时触发）、数据删除页（合规时触发）、轻量监控页（异常时触发）。
- 这些产品的共同特征：平时不需要，但需要的时候必须有，且用户愿意为此付费。

**不适合条件：**
- 需要大量定制和销售介入（→ R05）
- 责任风险高（→ R07 / R04）
- 付费意愿弱（→ R01）

**常见例子：**
- 政策页托管
- 状态页服务
- 产品更新日志 / Changelog
- 数据删除 / 合规页面
- 表单后端服务
- 邮件模板管理
- 轻量监控面板
- 代码片段管理

**推荐推进方式：**
- 1-3 个月出 MVP
- 标准定价，自助购买
- 做好 onboarding 和自助文档
- 关注续费率和流失率

## R03 · 内部基础设施 · Internal Infrastructure

**适合条件：**
- 复用价值高（S11 >= 4）
- 获客路径不清晰（S07 <= 2）
- 主要服务内部团队或自己的其他产品
- 不直接面向外部付费用户

**不适合条件：**
- 用户群体明确且外部需求强（→ R01 / R02）
- 维护成本极高（S10 >= 4）且没有内部团队支撑

**常见例子：**
- 内部 CI/CD 工具
- 内部数据面板
- 内部 API 网关
- 组件库 / 代码生成器

**推荐推进方式：**
- 先满足自己的需求
- 在内部验证后再考虑对外
- 如果外部需求出现，可升级为 R02 货架服务

---

## R04 · 第三方封装 / 集成层 · Third-party Wrapper

**适合条件：**
- 底层服务复杂、高风险（S08 >= 4 或 S09 >= 4）
- 存在成熟的第三方服务（S04 >= 4）
- 核心价值在于简化、整合、统一界面
- 不需要从零造底层

**不适合条件：**
- 第三方服务不存在或不成熟
- 封装层本身变成复杂系统
- 成本结构不可持续（第三方费用 > 收费）

**常见例子：**
- Vimeo/Mux/Wistia 嵌入管理
- Stripe/LemonSqueezy 包装面板
- Resend/SendGrid 邮件管理
- Notion/Google Docs API 封装
- AI API 聚合路由

**推荐推进方式：**
- 选成熟稳定的第三方服务
- 核心是简化配置、统一管理
- 不承诺底层服务的 SLA
- 明确标注依赖的第三方

## R05 · 销售型服务 · Sales-led Service

**适合条件：**
- 需要定制化接入
- 客单价高
- 客户需要人工服务、培训、集成支持
- S08（交付复杂度）>= 3 且有明确的付费客户

**不适合条件：**
- 可以标准化自助购买（→ R02）
- 单个开发者无法支撑销售和交付
- 没有明确的付费客户线索

**常见例子：**
- 企业内部工具定制
- 私有化部署
- 定制培训服务
- 集成咨询服务

**推荐推进方式：**
- 先找 1-2 个付费客户验证
- 确认客单价和交付成本
- 不要在没有付费信号时投入完整产品开发

---

## R06 · 内容或社区产品 · Content / Community Product

**适合条件：**
- 核心价值是内容、知识、社群关系
- 工具功能是辅助，不是核心
- S03（使用频率）中等或高

**不适合条件：**
- 核心价值是工具功能（→ R01 / R02）
- 需要高度复杂的后台系统

**常见例子：**
- 技术博客 / Newsletter
- 付费社群
- 教程 / 课程平台
- 论坛 / 讨论区
- 内容聚合

**推荐推进方式：**
- 先做内容，后做工具
- 验证受众和内容方向
- 不要在没有受众时造平台

---

## R07 · 高风险基础设施，暂缓 · High-risk Infrastructure, Defer

**适合条件（触发条件）：**
- S09（责任风险）>= 4 且 S08（交付复杂度）>= 4
- 涉及支付、身份认证、邮件送达、视频存储、合规、安全等

**路由到此类型时，可以建议：**
- 只读面板
- 第三方封装
- 配置管理界面
- 状态监控
- 轻量入口工具

**不可以建议：**
- 从零造完整支付网关
- 从零造完整邮件发送平台
- 从零造完整视频托管平台
- 从零造完整客服系统
- 从零造完整搜索引擎
- 从零造完整云部署平台

**常见例子（触发 R07 的想法）：**
- 完整视频托管平台
- 完整支付处理系统
- 完整邮件发送平台
- 完整客服工单系统
- 完整 CI/CD 平台

**推荐推进方式：**
- 窄化范围，只做管理/配置层
- 使用现有第三方服务作为底层
- 如果窄化后可行，副路由建议 R04

---

## R08 · 只适合做功能，不适合独立产品 · Feature Only, Not Standalone

**适合条件：**
- 作为独立产品用户感知价值不足
- 但作为现有产品的功能模块非常有价值
- S02（痛点强度）中等，但不足以独立支撑一个产品
- S05（付费意愿）低于 3 — 用户不会单独为此付费

**不适合条件：**
- 用户痛点足够强，愿意独立付费（→ R01 / R02）
- 市场规模足够大，独立有意义

**常见例子：**
- 某个 SaaS 的「导出 PDF」功能
- 某个平台的「数据导入」功能
- 某个工具的「快捷键配置」功能
- 某个 App 的「暗色模式」功能

**推荐推进方式：**
- 作为现有产品的功能模块实现
- 不要独立建站、独立推广
- 可作为开源模块贡献

## R09 · 信息不足，先调研 · Research First

**触发条件（满足任一即触发）：**
- 关键维度中有 3 个以上证据等级为 D
- S06（用户群体明确度）<= 2
- S01（触发场景明确度）<= 2
- 多数关键维度的置信度为 C 或 D

**关键维度：**
- S01 触发场景明确度
- S02 痛点强度
- S05 付费意愿证据
- S06 用户群体明确度
- S07 获客路径清晰度

**推荐推进方式：**
- 先完成 3 个最低成本验证动作
- 重新收集证据后再评估
- 不要在没有证据的情况下做完整产品

## 硬性路由规则

以下规则具有最高优先级，不可被其他考虑覆盖：

1. **如果 S09 >= 4 且 S08 >= 4：**
   → 主路由 R07（高风险基础设施，暂缓）
   → 副路由可选 R04（第三方封装）

2. **如果 S02 >= 4 且 S03 <= 2 且 S09 <= 3：**
   → 主路由 R02（货架服务）
   — 强痛点 + 低频使用 = 适合标准化 SaaS

3. **如果 S03 >= 4 且 S02 <= 3 且 S05 <= 2：**
   → 主路由 R01（入口工具）
   — 高频使用 + 弱痛点 + 弱付费 = 适合免费入口工具

4. **如果 S11 >= 4 且 S07 <= 2：**
   → 主路由 R03（内部基础设施）
   — 高复用 + 获客不清晰 = 适合内部使用

5. **如果 S06 <= 2：**
   → 主路由 R09（信息不足，先调研）
   — 不知道用户是谁 = 没法做判断

6. **如果 S01 <= 2：**
   → 主路由 R09（信息不足，先调研）
   — 不知道什么场景触发 = 没法做判断

7. **如果 S05 <= 2 且 S07 >= 4：**
   → 主路由 R01（入口工具）
   — 弱付费 + 清晰获客 = 做免费入口，靠流量

8. **如果 S04 >= 4 且 S05 >= 3 且 S08 <= 3：**
   → 可以考虑 R02（货架服务）
   — 成熟市场 + 有付费意愿 + 可交付 = 标准化 SaaS

9. **如果多数关键维度置信度为 C/D：**
   → 主路由 R09（信息不足，先调研）
   — 证据太弱 = 先收集证据

## 低频强痛点例外规则

不得因为 S03 使用频率低，就自动判定产品价值低。

如果同时满足以下条件：

- S03 <= 2（使用频率低）
- S01 >= 4（触发场景明确）
- S02 >= 4（痛点强度高）
- S09 <= 3（责任风险可控）

则应优先考虑 **R02 货架服务**。

**逻辑说明：**

低频不等于不重要。某些产品的价值恰恰在于「平时不需要，但需要的时候必须有」。典型场景包括：

- App 提交审核时需要隐私政策 URL（S03 低，S01 高，S02 高）
- 服务故障时需要状态页（S03 低，S01 高，S02 高）
- 合规检查时需要数据删除页（S03 低，S01 高，S02 高）
- 产品发布时需要更新日志页（S03 低，S01 高，S02 高）

这些产品的共同特征：
1. 触发事件是外部驱动的（审核、故障、合规、发布），不是用户日常习惯
2. 触发时如果没有解决方案，后果严重（审核被拒、客户流失、合规违规）
3. 用户愿意为「需要时有」付费（货架服务的核心逻辑）

**此规则不可用于：**
- S09 >= 4 的高风险产品（仍然走 R07）
- S01 <= 2 的场景模糊产品（仍然走 R09）
- 纯粹因为「我觉得低频也可以」而无证据支持的情况

## 一次评估只允许一个主路由，最多一个副路由
