# 风险门槛（Risk Gates）

## 总则

当 S09（责任风险）>= 4 且 S08（交付复杂度）>= 4 时，必须触发风险门槛检查。这意味着该产品涉及高风险基础设施，不可推荐个人开发者或小团队立即做完整产品。

触发后，可以建议窄化方案（只读面板、配置管理、监控面板、第三方封装），不得建议从零建造完整的高风险系统。

---

## G01 · 支付与资金风险 · Payment / Money Risk Gate

**触发条件：** 产品涉及处理支付、存储支付信息、计算资金、管理账户余额。

**风险说明：**
- PCI-DSS 合规要求
- 支付故障 = 直接资金损失
- 欺诈、退款、争议处理
- 多币种、税务复杂性

**窄化建议：**
- 使用 Stripe / LemonSqueezy / Paddle 等成熟支付服务
- 不要自己存储支付信息
- 做支付管理面板（只读）而不是支付处理系统
- 做发票管理工具而不是支付网关

**绝对不要：**
- 从零造支付网关
- 自己存储信用卡信息
- 自己处理支付清算

---

## G02 · 身份认证风险 · Auth / Identity Risk Gate

**触发条件：** 产品涉及用户注册、登录、权限管理、OAuth 集成、SSO。

**风险说明：**
- 密码存储和加密
- Session / Token 安全
- OAuth 集成的复杂性
- 多租户权限隔离
- 账户恢复流程

**窄化建议：**
- 使用 Auth0 / Clerk / Supabase Auth / NextAuth 等成熟方案
- 做权限配置面板而不是认证系统
- 做 SSO 配置管理而不是身份提供商

**绝对不要：**
- 从零造密码哈希和存储系统
- 从零造 OAuth 服务器
- 从零造多租户权限系统（如果这是核心基础设施）

---

## G03 · 邮件送达风险 · Email Deliverability Risk Gate

**触发条件：** 产品涉及发送事务邮件、营销邮件、批量邮件。

**风险说明：**
- 域名信誉管理
- 邮件送达率（deliverability）
- 退信率、投诉率
- SPF / DKIM / DMARC 配置
- IP 预热

**窄化建议：**
- 使用 Resend / SendGrid / AWS SES / Postmark 等成熟服务
- 做邮件模板管理而不是邮件发送平台
- 做发送状态监控面板

**绝对不要：**
- 从零造邮件发送平台
- 自己管理 SMTP 服务器（如果这是核心功能）
- 忽略域名信誉和送达率问题

---

## G04 · 隐私与合规风险 · Privacy / Compliance Risk Gate

**触发条件：** 产品涉及收集用户数据、存储个人信息、跨地区数据传输、行业合规。

**风险说明：**
- GDPR / CCPA / 个人信息保护法
- 数据跨境传输
- 用户数据删除权
- Cookie 同意
- 行业特定合规（HIPAA、SOC 2、PCI 等）

**窄化建议：**
- 做数据管理面板而不是数据处理系统
- 使用合规的第三方存储方案
- 在 MVP 阶段明确不服务受监管行业
- 做隐私政策生成器而不是合规保证工具

**绝对不要：**
- 承诺合规兜底
- 在没有法律顾问的情况下处理敏感个人数据
- 忽略用户数据删除和导出要求

---

## G05 · 视频、存储、带宽风险 · Video / Storage / Bandwidth Risk Gate

**触发条件：** 产品涉及视频托管、大文件存储、高带宽内容分发。

**风险说明：**
- 存储成本随使用量线性增长
- 带宽成本可能失控
- 视频转码的计算成本
- 版权和内容审核
- 滥用（CSAM、盗版、违规内容）

**窄化建议：**
- 使用 Vimeo / Mux / Wistia / Cloudflare Stream 等成熟服务
- 做视频管理面板（嵌入管理、课程目录）而不是视频托管平台
- 做存储配置管理而不是存储系统
- 设置成本上限和自动熔断

**绝对不要：**
- 从零造视频托管平台
- 从零造视频转码系统
- 忽略内容审核和版权风险
- 不做成本上限规划

---

## G06 · 核心通知风险 · Core Notification Risk Gate

**触发条件：** 产品依赖实时通知（Push、WebSocket、邮件、短信）作为核心功能。

**风险说明：**
- 推送通知的平台依赖（APNs、FCM）
- WebSocket 连接管理和扩展
- 短信送达率和成本
- 通知送达的可靠性直接影响核心体验

**窄化建议：**
- 使用 Pusher / Ably / Firebase / OneSignal 等成熟服务
- 做通知配置管理而不是通知发送系统
- 做通知状态监控面板

**绝对不要：**
- 从零造实时推送系统
- 低估通知送达率对用户体验的影响

---

## G07 · 客服可靠性风险 · Customer Support Reliability Risk Gate

**触发条件：** 产品涉及客服系统、工单系统、实时聊天、Help Desk。

**风险说明：**
- 客服系统直接影响客户满意度和留存
- SLA 承诺和响应时间
- 多渠道整合（邮件、聊天、电话）
- 知识库和自动化

**窄化建议：**
- 使用 Crisp / Intercom / Zendesk / Help Scout 等成熟服务
- 做轻量消息收件箱（统一查看多个渠道的消息）
- 做常见问题页面和文档站点
- 做客服质量监控仪表板

**绝对不要：**
- 从零造完整 Intercom / Zendesk 替代品
- 承诺不切实际的 SLA

---

## G08 · 部署与基础设施 SLA 风险 · Deployment / Infrastructure SLA Risk Gate

**触发条件：** 产品需要高可用性、自动扩缩容、多区域部署。

**风险说明：**
- 服务可用性承诺
- 数据备份和灾难恢复
- 自动扩缩容
- 多区域部署
- 监控和告警

**窄化建议：**
- 使用 Vercel / Railway / Fly.io / Cloudflare 等成熟平台
- 做部署配置面板而不是部署平台
- 做状态页和监控面板
- 在 MVP 阶段明确不承诺高可用 SLA

**绝对不要：**
- 从零造云部署平台
- 在 MVP 阶段承诺 99.9% 可用性
- 在没有运维经验的情况下做多区域部署

---

## 风险门槛检查输出格式

在评估报告的风险门槛章节中，按以下格式输出：

```
## 风险门槛

### 命中的风险门槛

| 门槛 ID | 名称 | 触发原因 | 窄化建议 |
|---|---|---|---|
| G05 | 视频、存储、带宽风险 | 涉及视频托管和转码 | 使用 Vimeo/Mux 嵌入管理 |

### 未命中的风险门槛

（列出检查但未命中的门槛，简要说明为什么未命中）
```
