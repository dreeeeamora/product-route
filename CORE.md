# CORE.md — Product Route 不可变核心

> 本文件定义 Product Route 的不可变核心。任何扩展、翻译、集成必须兼容本文件中的原则和流程。

---

## 唯一任务

**根据证据，把一个产品想法路由到合适的推进模式。**

---

## 它不负责

以下不是 Product Route 的职责范围。任何 Skill 变体不得将这些作为核心功能：

- 生成创业点子
- 鼓励用户开发
- 写商业计划书
- 预测产品一定成功或失败
- 替代法律、财务、投资建议
- 评估市场规模或 TAM
- 判断想法「好不好」

---

## 固定流程（不可跳过）

每次评估必须按以下顺序执行，不可跳过任何步骤：

1. **复述想法** — 用一句话复述用户的产品想法，确认理解一致
2. **收集证据** — 建立证据表，每条证据标注类型、来源、置信度
3. **找反证** — 主动寻找和列出反面证据、风险信号
4. **12 维评分** — 按照 S01-S12 维度逐项评分
5. **证据置信度惩罚** — 对证据不足的维度进行分数限制
6. **风险门槛判断** — 检查 G01-G08 风险门槛
7. **输出主路由和副路由** — 基于分数、惩罚、风险门槛综合判断
8. **定义 MVP 边界** — 明确第一版做什么、不做什么
9. **输出最低成本验证动作** — 3 个具体可执行动作
10. **输出结构化 JSON** — 完整的机器可读结果

---

## 核心原则

### P01：先证据，后评分

不得在没有证据的情况下给出分数。如果证据不足，必须在评分中标注，并通过置信度惩罚降低分数。

### P02：没证据就降置信度

任何评分维度如果没有 A 级或 B 级证据支撑，该维度分数不得高于 3 分。如果只有 D 级证据，不得高于 2 分。

### P03：不能只用总分决策

路由判断必须基于维度组合和规则匹配，不得仅凭总分高低做出路由决策。

### P04：高风险基础设施必须先过风险门槛

如果 S09（责任风险）>= 4 且 S08（交付复杂度）>= 4，必须触发风险门槛检查，不得推荐立即做完整产品。

### P05：一次评估只能有一个主路由，最多一个副路由

主路由是核心推荐，副路由是备选方案或辅助模式。不得输出 3 个以上的路由。

### P06：商业扩展只能增强证据覆盖，不能成为核心依赖

外部 Provider、MCP 工具、案例库等只能用于提高证据覆盖率和置信度。核心 Skill 必须能在无外部服务的情况下独立运行。

## 不可变 ID 体系

以下 ID 体系是 Product Route 的稳定接口，翻译和扩展不可改变：

| 前缀 | 范围 | 用途 |
|---|---|---|
| S01-S12 | scoring-rules | 评分维度 |
| R01-R09 | routing-rules | 路由类型 |
| G01-G08 | risk-gates | 风险门槛 |
| P01-P06 | CORE.md | 核心原则 |
| D001-D004 | DECISIONS.md | 架构决策 |

## 版本兼容性

任何兼容 Product Route 的实现必须：

1. 遵循固定流程的 10 个步骤
2. 使用稳定的 S/R/G ID 体系
3. 遵守 P01-P06 核心原则
4. 输出符合 result.schema.json 的 JSON
