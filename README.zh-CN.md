# Product Route · 产品路线判断 Skill

> **一个决策路由框架：把产品想法映射到正确的执行模式。**
>
> 大多数产品失败不是因为想法差，而是执行模式错了。

[English](README.en.md) | [项目主页](README.md)

---

## 这是什么

Product Route 只回答一个问题：

> 根据当前证据，这个想法应该用什么模式推进？

它不告诉你「这个想法好不好」。它告诉你「这个想法应该怎么执行」。

| 如果你的想法是… | Product Route 告诉你… |
|---|---|
| 二维码生成器 | 不要做 SaaS。做入口工具。 |
| 视频托管平台 | 不要造基础设施。封装 Mux/Vimeo。 |
| 隐私政策工具 | 不要做免费工具。做货架服务。 |
| AI Twitter 摘要 | 不要做工具。做内容+自动化输出。 |

## 这不是什么

- ❌ 想法打分器（"你的想法 7.3 分"）
- ❌ 商业计划书生成器
- ❌ 创业验证器
- ❌ 投资建议

## 问题出在哪里

**传统的产品思维在问错误的问题。**

问"这个想法好不好"没有意义，因为：
- 大多数想法既不好也不差 — 它们只是未被验证
- 同一个想法，用不同执行模式，结果天差地别
- "好想法 + 错模式"杀死的项目比"坏想法"多得多

**例子：** 一个隐私政策生成器。
- SaaS 模式 → 难差异化，付费意愿弱 → 大概率失败
- 货架服务模式 → 低维护，明确触发（App 审核）→ 可持续
- *同一个想法，不同执行模式，完全相反的结果。*

**执行模式就是产品本身。** 不是事后补充 — 它是核心决策。

## 工作流程

```
输入                      处理                        输出
┌──────────────┐         ┌───────────────────┐         ┌───────────────────┐
│ 想法          │         │ 证据检查          │        │ 路由 (R01-R09)    │
│ 约束          │  ──▶   │ 风险门槛过滤       │  ──▶   │ MVP 边界          │
│ 上下文        │         │ 执行路由决策       │         │ 验证动作          │
└──────────────┘         └───────────────────┘         └───────────────────┘
```

**输入：** 想法描述、已知约束、上下文信号（用户数据、竞品信息、市场信号）

**处理流程：**
1. 证据提取 — 我们到底知道什么？
2. 12 维评分 — S01 触发场景明确度 到 S12 当前阶段适配度
3. 置信度惩罚 — 证据不足 → 分数封顶
4. 风险门槛检查 — G01 支付风险 到 G08 基础设施风险
5. 执行路由 — 硬性规则将评分映射到执行模式

**输出：**
- 执行模式（R01-R09 之一）
- MVP 边界（做什么、不做什么）
- 最低成本验证动作（1-3 天，具体步骤）

## 9 种执行模式

| ID | 名称 | 一句话含义 | 真实例子 |
|---|---|---|---|
| R01 | 入口工具 | 免费低门槛，靠分发获客 | 二维码生成器 |
| R02 | 货架服务 | 标准化 SaaS，用户自助购买 | 隐私政策托管 |
| R03 | 内部基础设施 | 为自己而做，复用价值 > 外部需求 | 内部 CI/CD 面板 |
| R04 | 第三方封装 | 简化底层复杂服务 | Stripe 支付管理面板 |
| R05 | 销售型服务 | 定制接入，高客单价 | 企业 SSO 集成 |
| R06 | 内容/社区 | 依赖分发，内容即产品 | AI Twitter 摘要 |
| R07 | 高风险暂缓 | 窄化范围，不要从零造 | 视频托管平台 |
| R08 | 只做功能 | 应该放在现有产品里 | "导出 PDF" 按钮 |
| R09 | 先调研 | 证据不够，先做 3 个验证动作 | "给开发者用的东西" |

## 完整示例

### 输入
> "做一个 AI 工具，自动摘要 Twitter 上的热门长线程、列表和书签。"

### Product Route 给出的判断：

**路由：R06 — 内容/社区产品**

为什么不是 SaaS？核心价值是摘要内容本身。用户为内容而来，不为工具而来。没有内容分发循环，留存会死。正确的执行模式是内容优先：先发摘要，攒受众，再叠加工具。

为什么不是 R01 入口工具？入口工具意味着工具本身就是产品。但这里工具只是内容生产的手段。

**MVP 边界：**
- 做：自动摘要生成 pipeline + 定时发布
- 不做：用户系统、付费层级、自定义 AI 模型、Dashboard

**验证动作（1 天）：**
1. 手动摘要 10 条热门 Twitter 长线程
2. 以"AI 辅助摘要"标注发布
3. 对比互动量和正常推文

## 反模式

常见的执行模式错误：

| 反模式 | 错误模式 | 正确模式 | 原因 |
|---|---|---|---|
| 把内容产品做成 SaaS | R02 | R06 | 分发问题，不是工具问题 |
| 把基础设施当独立产品 | R07 | R08/R04 | 一个人维护不了视频托管 |
| 把封装做成平台 | R04 | R02 | 封装 Stripe ≠ 做支付平台 |
| 把免费的东西收费 | R02 | R01 | 二维码永远不会是订阅生意 |
| 没证据就动手 | R01-R08 | R09 | "可能有人需要"不是证据 |

## Quick Start

### 使用方式 A：作为 Skill 包安装（推荐）

将 `skills/product-route.zh-CN/` 安装为 Skill 包。Agent 检测到产品想法评估、产品定位、MVP 边界、推进模式等意图时自动触发。

### 使用方式 B：单文件 Prompt Bundle

将 `dist/product-route.zh-CN.bundle.md` 作为 System Prompt 注入。

### 使用方式 C：程序化调用

参考 `schemas/result.schema.json`，将输出 JSON 接入产品决策 pipeline。
